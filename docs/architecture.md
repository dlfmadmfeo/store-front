# 아키텍처 메모

## 목표

이 프로젝트를 단순한 화면 구현물이 아니라, 대고객 서비스형 프론트엔드 구조로 설계하고 설명할 수 있는 상태로 만드는 것이 목표입니다.

## 핵심 원칙

1. 가능하면 UI 컴포넌트가 fixture 데이터를 직접 가지지 않는다.
2. 도메인 데이터는 아래 흐름을 따른다.
   `mock -> route handler -> api client -> hook/state -> page/component`
3. 공통 타입은 `src/lib/types`에 둔다.
4. 공통 비동기 상태 UI는 `src/components/common`에 둔다.

## 이렇게 나눈 이유

이 구조는 아래 장점이 있습니다.

- 데이터 흐름이 명확하다.
- 비동기 상태가 화면에 드러난다.
- 도메인 경계가 더 선명해진다.
- 나중에 실제 API로 교체할 때 비용이 줄어든다.

## 현재 반영 범위

- 마이쇼핑: 반영 완료
- 홈 오늘의 이벤트 영역: 반영 완료
- 인증 로그인/회원가입: route handler 기반 mock API로 정리
- 장바구니 초기 데이터 로딩: 반영 완료
- auth / cart / home / myshopping 기본 route handler: 추가 완료
- 검색 패널 / 검색 결과 / 카테고리 메뉴: 반영 완료
- 운영 도구 주문 관리 화면: 추가 완료

## 다음 확장 후보

- 상품 상세 도메인
- 결제 직전 플로우
- 운영 화면의 역할별 권한 분리

## 운영 화면을 따로 둔 이유

대고객 화면과 운영 도구는 같은 주문 데이터를 보더라도 요구사항이 다릅니다.

- 고객 화면은 구매 흐름과 상태 확인이 우선입니다.
- 운영 화면은 검색, 필터, 메모, 상태 변경처럼 관리 효율이 우선입니다.

이번에 추가한 `/ops/orders`는 이 차이를 분리해서 드러내기 위한 화면입니다.

- 고객용 주문 화면: `/myshop/orders`
- 운영용 주문 화면: `/ops/orders`

같은 mock 데이터를 쓰더라도 route handler와 api client를 분리해 두면, 나중에 실제 API가 붙었을 때 화면 책임을 유지하기 쉽습니다.

## 웹뷰 딥링크를 유틸로 분리한 이유

앱 웹뷰 진입 URL을 화면마다 문자열로 직접 만들기 시작하면 규칙이 쉽게 흩어집니다.

그래서 주문 화면 딥링크는 `src/lib/utils/myShoppingDeepLink.ts`에서 생성하도록 분리했습니다.

- 딥링크 규칙을 한 곳에서 관리할 수 있다.
- 운영 화면과 고객 화면이 같은 링크 규칙을 공유할 수 있다.
- 나중에 실제 앱 팀과 파라미터 계약을 맞출 때 수정 지점이 줄어든다.

## 성능 관점에서 정리한 기준

이번 리팩터링에서는 단순히 화면이 보이는 것보다, 어느 코드를 서버에서 처리하고 어느 코드를 클라이언트로 남길지 경계를 정리하는 데 집중했습니다.

- 검색 결과, 카테고리, 마이쇼핑 홈처럼 정적인 초기 데이터는 서버에서 먼저 렌더링한다.
- 주문 관리나 주문 검색처럼 상호작용이 많은 화면만 클라이언트 상태를 유지한다.
- 정적인 탐색 데이터는 캐시하고, 사용자 상태가 강한 데이터만 `no-store`로 둔다.
- 무거운 팝오버와 운영 화면은 동적 import로 나눠 초기 번들을 가볍게 유지한다.
